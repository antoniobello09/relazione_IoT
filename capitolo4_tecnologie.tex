\chapter{Tecnologie Utilizzate}
\label{ch:tecnologie}

In questo capitolo vengono descritte le principali tecnologie impiegate nella realizzazione di ChillChain, evidenziando per ciascuna gli aspetti specifici che hanno contribuito alla soluzione adottata. La scelta dello stack tecnologico è stata guidata dalla necessità di integrare in un'unica piattaforma componenti eterogenee --- un backend per la logica di business, un frontend interattivo, servizi di streaming e analisi dati, un broker per la comunicazione asincrona e un database orientato ai documenti --- garantendo al contempo semplicità di deployment e riproducibilità dell'ambiente di esecuzione.

\section{Spring Boot}

Spring Boot è il framework Java utilizzato per la realizzazione del backend della piattaforma. La scelta è ricaduta su questo framework per la sua capacità di semplificare la configurazione di applicazioni enterprise, eliminando gran parte del boilerplate tipico dello sviluppo con Spring tradizionale grazie al principio della \textit{convention over configuration}.

Nel contesto di ChillChain, Spring Boot è stato impiegato per esporre le API REST che gestiscono l'intero ciclo di vita delle entità di dominio: veicoli, tratte, spedizioni, notifiche e utenti. Il modulo Spring Security è stato configurato per implementare l'autenticazione basata su token JWT con sessioni stateless, definendo una catena di filtri che intercetta ogni richiesta HTTP e ne verifica l'autorizzazione in base al ruolo dell'utente (amministratore, cliente, tecnico). L'integrazione con MongoDB è stata realizzata tramite Spring Data MongoDB, che consente di definire repository dichiarativi con query derivate dal nome dei metodi e pipeline di aggregazione personalizzate --- queste ultime utilizzate, ad esempio, per calcolare il prezzo delle tratte condivise eseguendo \textit{join} tra collezioni e applicando la logica di divisione direttamente a livello di database. Un aspetto particolarmente rilevante è l'integrazione con il protocollo MQTT tramite Spring Integration: il backend sottoscrive i topic di telemetria e anomalie pubblicati dai servizi Python, riceve i messaggi attraverso channel adapter dedicati e li instrada a service activator che si occupano della persistenza dei dati e della generazione delle notifiche.

\section{React e TypeScript}

Il frontend della piattaforma è stato realizzato con React, una libreria JavaScript per la costruzione di interfacce utente basate su componenti, in combinazione con TypeScript, un superset tipizzato di JavaScript che aggiunge il controllo statico dei tipi.

L'impiego di React ha permesso di strutturare l'interfaccia come un insieme di componenti riutilizzabili, ciascuno responsabile di una porzione specifica della vista: la lista dei veicoli, il form di invio spedizione, la mappa delle tratte, la dashboard telemetrica e il pannello notifiche. La gestione dello stato applicativo si basa sugli hook nativi di React (\texttt{useState}, \texttt{useEffect}) e su \texttt{localStorage} per la persistenza delle informazioni di sessione attraverso la navigazione. TypeScript ha contribuito a ridurre gli errori a runtime fornendo un sistema di tipi che rende espliciti i contratti tra componenti, particolarmente utile nella gestione dei DTO provenienti dal backend. La visualizzazione dei dati telemetrici in tempo reale è stata realizzata con la libreria Recharts, che consente di costruire grafici ad area con aggiornamento dinamico e codifica cromatica in base allo stato di guasto. La visualizzazione cartografica delle tratte si appoggia a React-Leaflet, impiegato per il rendering delle polyline dei percorsi su mappa interattiva. L'intero progetto frontend utilizza Vite come build tool, scelto per la velocità del server di sviluppo e il supporto nativo al \textit{hot module replacement}, e Tailwind CSS come framework di utility per lo stile, che ha permesso di ottenere un'interfaccia coerente e responsive senza scrivere fogli di stile personalizzati.

\section{MQTT e Mosquitto}

MQTT (\textit{Message Queuing Telemetry Transport}) è il protocollo di comunicazione adottato per lo scambio di dati telemetrici tra i servizi della piattaforma. Si tratta di un protocollo di messaggistica leggero basato sul pattern \textit{publish/subscribe}, progettato specificamente per contesti IoT caratterizzati da banda limitata e risorse computazionali ridotte.

In ChillChain, MQTT costituisce il tessuto connettivo tra tre attori: il servizio di streaming (publisher), il servizio di anomaly detection (subscriber e publisher) e il backend Spring Boot (subscriber). Il broker Mosquitto, un'implementazione open source del protocollo MQTT, gestisce l'instradamento dei messaggi attraverso una struttura gerarchica di topic nella forma \texttt{fridge/\{vehicleName\}/telemetry} per i dati sensoristici e \texttt{fridge/\{vehicleName\}/anomalies} per i risultati dell'analisi. L'uso del carattere jolly \texttt{+} nelle sottoscrizioni consente a ciascun subscriber di ricevere i messaggi relativi a tutti i veicoli senza dover conoscere a priori i nomi delle sorgenti. Questa architettura disaccoppia completamente i produttori dai consumatori dei dati: il servizio di streaming pubblica i messaggi senza sapere chi li riceverà, e il servizio di anomaly detection può essere avviato, fermato o sostituito senza impattare sugli altri componenti. La riconnessione automatica, configurata sia lato Python (tramite la libreria Paho MQTT) sia lato Spring Boot (tramite le opzioni del client factory), garantisce la resilienza della comunicazione in caso di interruzioni temporanee del broker.

\section{MongoDB}

MongoDB è il database NoSQL orientato ai documenti impiegato per la persistenza di tutti i dati della piattaforma. A differenza dei database relazionali, MongoDB memorizza i dati in documenti JSON-like (BSON) organizzati in collezioni, senza imporre uno schema rigido.

Nel contesto di ChillChain, questa flessibilità si è rivelata particolarmente vantaggiosa per la gestione dei dati telemetrici, la cui struttura può variare in funzione del tipo di sensore, e per le entità di dominio come tratte e spedizioni, che includono campi con struttura annidata (coordinate geografiche, polyline). Il sistema utilizza sei collezioni principali: \texttt{vehicle}, \texttt{trip}, \texttt{shipment}, \texttt{notification}, \texttt{telemetry} e \texttt{user}. L'interazione avviene tramite Spring Data MongoDB, che mappa automaticamente le classi Java alle collezioni corrispondenti. Un aspetto specifico del progetto è l'uso delle pipeline di aggregazione di MongoDB per implementare la logica di ricerca delle tratte condivise: una singola query esegue \textit{lookup} tra le collezioni \texttt{trip}, \texttt{vehicle} e \texttt{shipment}, filtra per capacità residua, tipologia di refrigerazione e data di arrivo, e calcola il prezzo scontato dividendo il costo minimo per il numero di spedizioni, il tutto a livello di database senza necessità di elaborazione applicativa. Gli indici geospaziali sulle coordinate di partenza e arrivo delle tratte e delle spedizioni consentono inoltre di eseguire query di prossimità per future estensioni del sistema.

\section{Python, FastAPI e PyTorch}

Python è il linguaggio utilizzato per i due microservizi dedicati rispettivamente allo streaming dei dati telemetrici e alla rilevazione delle anomalie. La scelta è motivata dalla ricchezza dell'ecosistema Python per il \textit{data processing} e il \textit{machine learning}, che ha consentito di implementare entrambi i servizi con un numero ridotto di dipendenze.

Il servizio di streaming (\texttt{streamer\_main.py}) è un'applicazione FastAPI che legge i file CSV contenenti i dati simulati di telemetria e li pubblica sul broker MQTT con una cadenza configurabile, simulando il comportamento di sensori reali installati a bordo dei veicoli. I dataset sono generati da uno script di simulazione termodinamica (\texttt{stream\_fridge\_dataset\_generator.py}), derivato dal dataset Kaggle \textit{Simulated Refrigerator Fault Diagnosis} \cite{kaggle_fridge}, che modella il comportamento fisico di un impianto frigorifero a compressione di vapore includendo scambi termici con l'ambiente, cicli di sbrinamento, aperture della porta e tredici tipologie di guasto iniettabili con parametri stocastici. Questo approccio consente di generare scenari realistici di funzionamento normale e anomalo con pieno controllo sulle condizioni sperimentali.

Il servizio di anomaly detection (\texttt{anomaly\_main.py}) è anch'esso un'applicazione FastAPI che sottoscrive il topic telemetria via MQTT, analizza i dati in ingresso e pubblica i risultati sul topic anomalie. Il cuore del servizio è un modello LSTM Autoencoder implementato in PyTorch: l'encoder, una rete LSTM, comprime la sequenza temporale di ingresso in una rappresentazione latente a dimensionalità ridotta; il decoder, un'altra rete LSTM, ricostruisce la sequenza originale a partire da tale rappresentazione. L'errore di ricostruzione, calcolato come errore quadratico medio (MSE) tra input e output, viene confrontato con una soglia appresa durante l'addestramento su dati di funzionamento normale. Lo scaler, serializzato con Pickle, normalizza le quindici feature in ingresso per garantire che parametri con scale numeriche molto diverse (temperature in gradi, pressioni in bar, potenze in watt) contribuiscano in modo equilibrato all'errore di ricostruzione. La libreria Pandas è impiegata per la lettura dei CSV e la preparazione dei DataFrame, mentre NumPy gestisce le operazioni numeriche sui buffer a finestra scorrevole.

\section{Google Maps Platform}

Le API di Google Maps Platform sono utilizzate per il calcolo dei percorsi e la geolocalizzazione degli indirizzi inseriti dagli utenti. In particolare, il servizio di backend impiega due API distinte: la \textit{Geocoding API}, che converte gli indirizzi testuali in coordinate geografiche (latitudine e longitudine), e la \textit{Routes API}, che calcola il percorso ottimale tra due o più punti restituendo la distanza in metri, la durata stimata e la polyline codificata del tracciato.

Nel flusso operativo di ChillChain, queste API vengono invocate in due momenti: alla creazione di una nuova tratta da parte dell'amministratore, per calcolare il percorso base tra partenza e arrivo, e all'inserimento di una nuova spedizione condivisa, per ricalcolare il percorso con i waypoint intermedi corrispondenti ai punti di ritiro e consegna del nuovo pacco. La polyline restituita dall'API viene memorizzata nel database e trasmessa al frontend, dove viene decodificata e renderizzata su mappa tramite React-Leaflet. L'invocazione delle API avviene lato server tramite \texttt{RestTemplate} di Spring, evitando di esporre la chiave API al client.

\section{Docker e Docker Compose}

Docker è la piattaforma di containerizzazione impiegata per isolare ciascun componente dell'architettura in un ambiente di esecuzione autonomo e riproducibile. Ogni servizio della piattaforma --- il backend Spring Boot, il frontend React, il broker Mosquitto, il servizio di streaming Python, il servizio di anomaly detection e il database MongoDB --- viene eseguito all'interno di un container dedicato, con le proprie dipendenze e la propria configurazione.

Docker Compose orchestra l'avvio coordinato di tutti i container, definendo in un unico file YAML le immagini, le porte esposte, le variabili d'ambiente, i volumi persistenti e le reti interne. La rete Docker interna consente ai servizi di comunicarsi utilizzando i nomi dei container come hostname (ad esempio, il backend raggiunge MongoDB tramite l'hostname \texttt{mongodb} e il broker MQTT tramite \texttt{mosquitto}), eliminando la necessità di configurare indirizzi IP statici. Questa scelta garantisce che l'intero sistema possa essere avviato con un singolo comando \texttt{docker-compose up}, riproducendo un ambiente identico su qualsiasi macchina dotata del runtime Docker, indipendentemente dal sistema operativo sottostante.
